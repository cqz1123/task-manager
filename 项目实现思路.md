# 项目概述

**Task Manager** 是一款基于 Web 的任务管理协作工具，灵感来源于 Trello。该项目实现了完整的看板协作功能，支持多人协同管理任务，具备角色权限控制、实时同步等特性。

## 核心功能

| 功能模块 | 描述                                                    |
| -------- | ------------------------------------------------------- |
| 用户认证 | 注册、登录、JWT 令牌认证                                |
| 看板管理 | 创建、编辑、删除看板，邀请码分享                        |
| 列表管理 | 添加、编辑、删除任务列表                                |
| 卡片管理 | 创建、编辑、删除卡片，支持拖拽排序                      |
| 成员协作 | 通过邀请码加入看板，角色权限控制（owner/editor/viewer） |
| 个人设置 | 修改用户名、修改密码                                    |

## 开发过程记录（按模块）

### 1. 项目初始化与基础搭建

**问题**：初期对项目结构不清晰，搭建进度慢；Git 使用不熟练。  
**解决**：参考常见全栈项目结构，确立 `backend/` 和 `frontend/` 分离的组织方式；后续逐步熟悉 Git 命令。  
**数据库命名踩坑**：数据库名使用连字符（`task-manager`）导致 SQL 语句必须用反引号包裹，后改为下划线 `task_manager`。

### 2. 后端连接与认证模块

**实现**：使用 `mysql2/promise` 连接池，JWT + bcrypt 实现注册/登录。  
**注意事项**：  
token 恢复与用户状态需一致，避免有 token 但 user 为 null。  
axios 拦截器处理 401 时，使用 `router.push` 而非 `window.location.href` 避免页面刷新。  
token 过期处理：简单方案是自动登出，后续可扩展自动刷新。

### 3. 前端组件与状态管理

**类型理解**  
`import type { User } from '../api/auth'` 表示仅导入类型，不导入值，用于编译期类型检查。  
`defineProps<{ list: ListWithCards }>()` 是 Vue3 的运行时声明方式。

**弹窗管理优化**  
**问题**：编辑卡片弹窗定义在 `CardItem.vue` 中，每个卡片一个弹窗，导致显示位置错误、频繁闪烁。  
**解决**：将弹窗移到 `BoardDetail.vue` 统一管理，通过 `card-click` 事件传递选中的卡片。

**API 响应处理错误**  
**现象**：创建新卡片后不显示，切回再进入才出现；删除按钮无反应。  
**原因**：前端 API 未正确提取后端统一响应格式 `{ success: boolean, data: any }`。  
**解决**：修改所有 API 文件（`card.ts`, `list.ts`, `board.ts` 等），正确返回 `response.data.data`。

**导航栏功能缺失**  
**问题**：缺少“返回我的看板”按钮及当前看板名称。  
**解决**：导航栏组件预留插槽，在 `BoardDetail` 中传入看板名称和返回按钮。

**Element Plus 图标使用**  
删除按钮不显示，查阅文档后正确使用 `:icon="Delete"` 解决。

### 4. 拖拽功能实现

**乐观更新 + 失败回滚**：先改 UI，后发请求，保证流畅体验；请求失败时恢复旧数据。  

**跨列表拖拽**  
原先只实现了同列表排序，跨列表未完成。  
原因：错误绑定了原生 `@dragend` 事件，应使用 `vuedraggable` 的 `@end` 事件。  
解决：统一在 `BoardDetail` 监听 `@end`，通过 `closest('[data-list-id]')` 获取源/目标列表 ID。  

**空列表拖拽失败**：空列表的 draggable 容器高度太小（10px），设 `min-height: 100px` 解决。  

**数据库持久化缺失**  
拖拽后前端更新了，但数据库未更新，刷新页面恢复原状。  
解决：在 `moveCard` 中乐观更新后调用 `PATCH /api/cards/:cardId/position` 更新 `list_id` 和 `order_index`。

### 5. 权限系统与联合管理

**需求**：创建者拥有完整权限，可邀请他人；成员分为 `editor`（可编辑）和 `viewer`（只读）。  

**实现**：  
数据库新增 `board_members` 表记录 `(board_id, user_id, role)`。  
后端增加 `checkBoardRole` 中间件，验证用户对看板的角色。  
修改 `GET /api/boards` 返回每个看板的 `myRole` 字段。  
前端根据角色控制 UI（隐藏写操作、禁用拖拽等）。  

**成员管理**：仅 owner 可见成员管理弹窗，可查看成员、修改角色、移除成员。  

**邀请机制**：每个看板生成唯一邀请码，用户输入后加入（默认 `viewer`），再由 owner 调整角色。

### 6. 实时同步（WebSocket）

**目标**：多人编辑同一看板时，其他用户自动看到变化（新增卡片、拖拽移动等）。  

**实现**：  
后端使用 `socket.io`，与 Express 共享 HTTP 服务器，通过 `handshake.auth.token` 认证。  
前端连接后发送 `join-board` 加入房间 `board:${boardId}`。  
卡片创建、更新、删除、移动等 API 成功后调用 `emitToBoard` 广播事件。  

**问题与权衡**：  
广播后操作者自己会收到两次更新（乐观更新 + 广播）。简单处理：广播给所有人，前端接受幂等更新（重复移动不会出错）。  
列表的实时更新未完全实现，后续补充。

### 7. 任务完成（归档）功能

**需求**：点击“完成”将卡片移动到专用“已完成”列表，记录原列表 ID 以便撤销。  

**设计**：  
`cards` 表增加 `completed_from_list_id` 字段。  
`lists` 表增加 `is_system` 标识统列表。  
后端实现 `completeCard` 和 `uncompleteCard` API，使用事务维护顺序。  

**撤销逻辑**：利用 `completed_from_list_id` 将卡片移回原列表（若原列表已删除则移到第一个普通列表）。

### 8. 个人设置模块

**功能**：修改用户名、修改密码。  
**后端 API**：`PUT /api/user/profile` 和 `PUT /api/user/password`。  
**前端**：`NavBar` 中添加“个人设置”下拉菜单，使用 `UserSettings` 弹窗。  

**问题修复**：  
修改用户名后导航栏不更新：原因是直接赋值 `userStore.user = response.data` 破坏响应式，改为只更新 `user.username`。  
响应拦截器数据结构不匹配：统一使用 `response.success` 判断。

### 9. Bug 修复与体验优化

| 问题               | 解决方案                                        |
| ------------------ | ----------------------------------------------- |
| 滚动条多余         | 使用 `overflow: hidden`                         |
| 卡片内容溢出       | `word-break: break-word` + `-webkit-line-clamp` |
| 日期未汉化         | 引入 Element Plus 中文包 `zhCn`                 |
| 卡片字体偏小       | 增大卡片内字体及间距                            |
| 删除按钮误跳转     | 添加 `@click.stop` 阻止冒泡                     |
| 删除看板外键约束   | 改为 `ON DELETE CASCADE` 或事务级联删除         |
| 验证码刷新失效     | 修复前端刷新逻辑                                |
| 导航栏用户名不显示 | 修复响应式更新                                  |

### 10. 后端架构优化

原本控制器直接操作数据库，后重构为分层架构：  
`Controllers → Services → Database`  
控制器仅处理 HTTP 请求/响应及权限验证，业务逻辑下沉到 Service 层。

## 数据库设计

### 主要表结构

`users`：用户信息（id, username, email, password_hash ...）  
`boards`：看板（id, name, color, owner_id, invite_code ...）  
`lists`：列表（id, board_id, title, order_index, is_system）  
`cards`：卡片（id, list_id, title, description, due_date, assignee, order_index, completed_from_list_id）  
`board_members`：成员关系（board_id, user_id, role）  
`activities`：活动日志（可选）  
`card_labels`：标签（可选）

### ER 图（文字描述）

`users` 1 : N `boards`（owner）  
`boards` 1 : N `lists`  
`lists` 1 : N `cards`  
`boards` N : M `users`（通过 `board_members`，带角色）  
`cards` 可选指向 `lists`（`completed_from_list_id`）

详细建表 SQL 见仓库 `/database/schema.sql`。

## 核心 API 端点

| 认证 | POST   | `/api/auth/register`                    | 注册                       |

| 列表 | POST   | `/api/lists`                            | 创建列表                   |
| 列表 | PUT    | `/api/lists/:id`                        | 更新列表标题               |

| 卡片 | POST   | `/api/cards`                            | 创建卡片                   |

| 成员 | GET    | `/api/boards/:boardId/members`          | 获取成员列表               |
| 成员 | POST   | `/api/boards/:boardId/members`          | 添加成员                   |
| 成员 | PATCH  | `/api/boards/:boardId/members/:userId`  | 修改成员角色               |
| 成员 | DELETE | `/api/boards/:boardId/members/:userId`  | 移除成员                   |
| 用户 | PUT    | `/api/user/profile`                     | 修改用户名                 |
| 用户 | PUT    | `/api/user/password`                    | 修改密码                   |
| 邀请 | POST   | `/api/boards/join-by-code`              | 通过邀请码加入看板         |

Node.js 20+，MySQL 8+，pnpm / npm / yarn

```bash
cd backend
npm install
```
